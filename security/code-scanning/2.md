## What CodeQL reported

This is the same allocation as finding [#1](https://github.com/bitwise-media-group/evolve/security/code-scanning/1), on
the same line:

```go
merged := make([]Assertion, 0, len(e.Expectations)+len(e.Assertions))
```

CodeQL emits one alert per operand of the `+`: #1 flags `len(e.Expectations)` (cols 35–54) and this alert flags
`len(e.Assertions)` (cols 55–72). The "potentially large value" is traced to the JSON produced from a decoded spec file
at `internal/encfmt/encfmt.go:55` in both.

## Why this is a false positive

The disposition is identical to finding #1 — the full reasoning is in [1.md](./1.md). In short: both `e.Expectations`
and `e.Assertions` are slices `json.Unmarshal` has already materialized in memory before line 188, so the `make` merely
pre-sizes the concatenation of two existing slices rather than allocating from an attacker-chosen count. On evolve's
64-bit-only build targets the sum of two in-memory slice lengths cannot reach 2^63 without tens of exabytes of resident
RAM, so the addition cannot overflow.

## Resolution

No code change is warranted; see [1.md](./1.md) for the full analysis. This alert is dismissed in the code-scanning UI
as **false positive** alongside finding #1.
