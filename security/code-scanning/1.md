# Code scanning finding #1 — Size computation for allocation may overflow

**Finding:**
[github.com/bitwise-media-group/evolve/security/code-scanning/1](https://github.com/bitwise-media-group/evolve/security/code-scanning/1)

| Field    | Value                                                                               |
| -------- | ----------------------------------------------------------------------------------- |
| Rule     | `go/allocation-size-overflow` (CWE-190)                                             |
| Tool     | CodeQL 2.25.6                                                                       |
| Severity | Warning (security-severity: high)                                                   |
| Location | `internal/evalspec/evalspec.go:188` (cols 35–54, the `len(e.Expectations)` operand) |
| Detected | 2026-06-13 against commit `73a49b3`                                                 |
| Outcome  | **False positive**                                                                  |

## What CodeQL reported

The query flagged the capacity argument of a `make` and traced one operand back to a "potentially large value" — the
JSON produced from a decoded spec file at `internal/encfmt/encfmt.go:55`:

```go
merged := make([]Assertion, 0, len(e.Expectations)+len(e.Assertions))
```

This is the [`(*EvalsFile).normalize`](../../internal/evalspec/evalspec.go) merge step: an eval's authored
`expectations` are expanded into leading `llm` assertions, and the slice is pre-sized to hold both groups before they
are appended. Alerts [#1](https://github.com/bitwise-media-group/evolve/security/code-scanning/1) and
[#2](https://github.com/bitwise-media-group/evolve/security/code-scanning/2) flag the two operands of the same `+` on
this one line — #1 is `len(e.Expectations)`, #2 is `len(e.Assertions)` (see [2.md](./2.md)). The analysis below covers
both.

## Why this is a false positive

**1. Both addends are lengths of slices already materialized in memory.**

`e.Expectations` (`[]string`) and `e.Assertions` (`[]Assertion`) are populated by `json.Unmarshal` inside
`encfmt.DecodeFile` _before_ line 188 runs. The `make` does not allocate from an attacker-chosen count — it pre-sizes
the concatenation of two slices that already coexist in memory, and the next three lines append exactly those elements
into `merged`. The capacity is by construction no larger than the number of elements the process already holds.

**2. The sum cannot overflow on any supported platform.**

evolve ships only 64-bit builds (linux, darwin, windows × amd64, arm64), where Go's `int` is 64-bit. For
`len(e.Expectations) + len(e.Assertions)` to reach 2^63 the process would already hold ~9.2×10¹⁸ elements in memory —
each a 16-byte string header or a multi-field `Assertion` struct, i.e. tens of exabytes of resident RAM. The inputs
cannot be decoded in the first place, so the addition that consumes their lengths cannot wrap.

**3. The taint premise does not hold.**

The "large value" CodeQL traces from `encfmt.go:55` is the decoded file content, whose size is bounded by the file
already read into memory and unmarshaled. There is no multiplication and no from-scratch allocation sized by an
attacker-controlled count — only `len()` of data that demonstrably already exists. The pattern the rule warns about (an
untrusted size flowing into an unguarded allocation) is absent.

## Resolution

No code change is warranted: the construct is a standard capacity hint for a slice merge and cannot overflow on any
platform evolve targets. Because CodeQL is run over first-party `internal/` source with no path filter — by design, so
our own code is analysed — there is no path-ignore escape and none is wanted. Both alerts (#1 and #2) flag the two
operands of this single expression and are dismissed in the code-scanning UI as **false positive**.
